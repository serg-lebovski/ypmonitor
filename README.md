# YPMon — мониторинг резервного копирования

Система из двух частей для контроля резервного копирования на множестве серверов:

- **Агент** — ставится на сервер, где выполняются бэкапы. Делает резервные копии PostgreSQL,
  архивирует файлы в ZIP, показывает логи архивации MSSQL и отправляет итоги на сервер.
- **Сервер** — центральная точка: принимает данные от агентов, показывает дашборд с состоянием
  всех клиентов и серверов, шлёт оповещения при проблемах.

Технологии: .NET 8, ASP.NET Core (Razor Pages), EF Core (SQLite или PostgreSQL).
Каждый компонент собирается в **один self-contained исполняемый файл** без внешних зависимостей
и устанавливается как служба Windows или демон systemd (Linux).

---

## Состав установщиков

После сборки в папке `dist/` появляются три готовых установщика:

| Папка | Что это | Установка |
|-------|---------|-----------|
| `dist/windows-server` | Сервер для Windows | `install-service.ps1` от администратора |
| `dist/windows-agent`  | Агент для Windows  | `install-service.ps1` от администратора |
| `dist/linux-server`   | Сервер для Linux   | `sudo ./install.sh` |

## Сборка

Требуется .NET 8 SDK.

```powershell
# Всё сразу
pwsh build/build-all.ps1

# По отдельности
pwsh build/build-all.ps1 -Target server-win
pwsh build/build-all.ps1 -Target agent-win
pwsh build/build-all.ps1 -Target server-linux
```

## Развёртывание сервера через Docker (рекомендуется для Linux + PostgreSQL)

Сервер упакован в Docker, подключается к внешнему PostgreSQL. Деплой на хосте:

```bash
# 1. Клонировать репозиторий (первый раз)
git clone https://github.com/serg-lebovski/ypmonitor.git
cd ypmonitor

# 2. Создать .env со строкой подключения к PostgreSQL (в git не попадает)
cp .env.example .env
nano .env        # впишите Host/Database/Username/Password

# 3. Запустить
docker compose up -d --build

# Обновление до новой версии:
git pull && docker compose up -d --build
```

`.env`:
```
YPMON_HTTP_PORT=8080
YPMON_DB_CONNSTRING=Host=10.10.20.25;Port=5432;Database=ypmon;Username=admin_yp;Password=********
```

Требования к PostgreSQL: пользователь должен иметь право создавать БД (`CREATEDB`) — таблицы и сама
база `ypmon` создаются автоматически при первом старте. Веб-интерфейс: `http://<хост>:8080/`,
при первом входе создаётся администратор. Логи: `docker compose logs -f`.

> Файл с паролем (`.env`) и артефакты сборки в репозиторий не коммитятся (см. `.gitignore`).
> Папка обновлений агента смонтирована как том `ypmon-updates` (`/app/agent-updates`).

## Установка сервера (без Docker)

### Windows
1. Скопируйте папку `dist/windows-server` на машину сервера.
2. При необходимости измените порт/БД в `appsettings.json`.
3. Запустите PowerShell **от администратора** и выполните `install-service.ps1`.
4. Откройте `http://<адрес-сервера>:8080/` и создайте администратора (мастер первого запуска).

### Linux
1. Скопируйте папку `dist/linux-server` на сервер.
2. `sudo ./install.sh` — создаст пользователя `ypmon`, развернёт в `/opt/ypmon`,
   установит и запустит службу `ypmon-server`.
3. Откройте `http://<адрес-сервера>:8080/` и создайте администратора.

Логи: `journalctl -u ypmon-server -f`

### Настройка сервера (`appsettings.json`)
```json
{
  "Server": { "HttpPort": 8080 },
  "Database": {
    "Provider": "sqlite",          // sqlite | postgres
    "ConnectionString": ""          // для postgres укажите строку подключения
  }
}
```
При `sqlite` база создаётся автоматически в `data/ypmon.db`. Параметры применяются после
перезапуска службы.

## Установка и настройка агента (Windows)

У агента **два режима** работы из одного и того же `Ypmon.Agent.exe`:

- **Двойной клик по exe** — открывается **окно настроек** (нативное приложение Windows) со всеми
  параметрами. Пока окно открыто, агент уже выполняет задания и шлёт отчёты в фоне.
- **Служба Windows** — тот же exe, установленный как служба `YpmonAgent`, работает в фоне без окна.

> **Безопасность.** Агент устанавливает только **исходящее** соединение к серверу и **не слушает
> ни одного порта**. Сервер не может передать агенту никаких команд — он лишь принимает отчёты.
> Управление службой и её заданиями выполняется **только локально через окно настроек** (`config.json`).
> Один и тот же системный объект синхронизации гарантирует, что задания выполняет лишь один процесс
> (служба ИЛИ открытое окно) — без дублирования бэкапов.

### Порядок настройки
1. Скопируйте папку `dist/windows-agent` на сервер, где делаются бэкапы.
2. Запустите `Ypmon.Agent.exe` (двойной клик) — откроется окно настроек.
3. Вкладка **Подключение**: адрес сервера YPMon и **API-ключ**.
   Ключ берётся в веб-интерфейсе сервера: *Клиенты → выбрать клиента → добавить сервер →
   «Данные для подключения агента»*.
4. Вкладка **Задания архивации** — единый список заданий двух типов (кнопки «+ Бэкап PostgreSQL»
   и «+ Архив папок/файлов»):
   - **Бэкап PostgreSQL** — хост/порт/пользователь/пароль. Нажмите **«Проверить соединение и получить
     список БД»** — при успехе базу можно **выбрать из выпадающего списка** (или вписать вручную).
     Папка для дампов, число копий, интервал. Использует `pg_dump` (путь — на вкладке «Подключение»).
   - **Архив папок/файлов** — список папок/файлов (в т.ч. **сетевых** `\\сервер\папка`) → ZIP.
     Для доступа к сетевым ресурсам укажите **учётную запись Windows** (логин/пароль) и нажмите
     **«Проверить доступ к папкам»**. Папка архивов тоже может быть сетевой.
5. Вкладка **Логи MSSQL** — папка с логами; агент определяет статус (ок/ошибка) и включает его в отчёт.
6. Вкладка **Отчёт** — таблица созданных архивов: задание, тип, файл, **дата создания**, размер.
7. Вкладка **Служба** — установка агента как службы Windows: **имя службы** и **учётная запись**
   (логин/пароль, пусто = LocalSystem). Кнопки установить/удалить/запустить/остановить (через UAC),
   а также автообновление и ручная проверка обновлений.
8. Вкладка **Статус** — связь с сервером, состояние заданий, диски.
9. Кнопка **«Справка»** — полное описание всех функций агента.
10. Нажмите **«Сохранить всё»** (запишет `config.json`). Кнопка **«Выполнить бэкап сейчас»** —
    локальный запуск всех включённых заданий.

Службу можно установить как из окна (вкладка «Служба»), так и скриптом `install-service.ps1`
**от администратора**. Служба читает тот же `config.json`.

### Retention (срок хранения архивов)
У каждого задания задаётся «Хранить копий N». Когда создаётся новый архив и общее число превышает N,
**автоматически удаляются самые старые**, пока не останется ровно N. Например, при N = 7 создание
8-го архива удалит самый старый. Значение 0 отключает автоудаление.

Агент по расписанию выполняет задания, применяет retention и каждые N секунд отправляет на сервер:
доступность БД, что архивируется, сколько копий, состояние дисков.

## Обновление агентов с сервера

Агент умеет сам обновляться:

1. Соберите новую версию агента (повысьте `<Version>` в `Ypmon.Agent.csproj`).
2. Положите новый `Ypmon.Agent.exe` на сервере в папку **`agent-updates`** (рядом с исполняемым файлом
   сервера; создаётся автоматически).
3. Каждый агент с включённым автообновлением раз в день обращается к серверу
   (`GET /api/agent/version` с его API-ключом), сравнивает версию и при наличии новой —
   скачивает (`/api/agent/download`) и устанавливает её с перезапуском службы.
4. Можно обновить и вручную: в окне агента вкладка **Служба → «Проверить обновления сейчас»**.

Эндпоинты обновления требуют корректный `X-Api-Key` (как у отчётов). Обновление инициирует **сам агент**
(скачивает по своему запросу) — сервер по-прежнему не устанавливает с агентом входящих соединений.

## Работа с веб-интерфейсом сервера

- **Дашборд** — клиенты и их серверы, статус (ок / предупреждение / ошибка / офлайн),
  метрики сверху. Клик по серверу — панель свойств справа: задания, копии, диски, кнопка
  «Запросить бэкап сейчас». Автообновление каждые 30 сек.
- **Клиенты** — добавление клиентов (групп серверов) и серверов внутри клиента
  (название, физический адрес, IP, описание), выдача/перевыпуск API-ключей.
- **Настройки** — параметры сервера (порт/адрес/БД), порог «офлайн», срок хранения истории,
  оповещения (Telegram / e-mail), управление пользователями и редактирование своего профиля
  (имя, логин, пароль).

### Роли пользователей
- **Администратор** — полный доступ.
- **Наблюдатель** — только просмотр.

## Дополнительные возможности (сверх ТЗ)

- 🔔 **Оповещения** в Telegram и по e-mail при переходе сервера в проблемное состояние и при уходе в офлайн.
- 🟢 **Детект офлайн** — если агент молчит дольше порога, сервер помечает его офлайн.
- 💽 **Мониторинг дисков** на машинах агентов (свободное место, % занятости).
- ▶️ **Ручной запуск** бэкапа с сервера («Бэкап сейчас») — команда передаётся агенту.
- 🗂️ **Группировка** серверов по клиентам, роли пользователей, автоочистка истории отчётов.
- 🔐 API-ключ на каждый сервер, перевыпуск ключа, PBKDF2-хеширование паролей.
- 🔌 **Проверка соединения с PostgreSQL** прямо из агента и выбор базы из списка.
- 🌐 **Доступ к сетевым папкам** (источникам и месту хранения архивов) по логину/паролю Windows.
- 🧰 **Установка службы из окна** агента (имя + учётная запись) и кнопка проверки доступа.
- ⬆️ **Самообновление агента** с сервера (ежедневно или вручную).

## Архитектура

```
Ypmon.sln
├─ src/Ypmon.Shared   — общие DTO/контракты (отчёт агента, статусы заданий)
├─ src/Ypmon.Server   — ASP.NET Core: Razor Pages (UI) + API /api/report (приём от агентов)
└─ src/Ypmon.Agent    — фоновые службы (бэкапы, отправка отчётов) + окно настроек (WinForms)
```

Протокол: агент шлёт `POST /api/report` с заголовком `X-Api-Key`. Соединение строго одностороннее —
сервер только подтверждает приём и **не передаёт агенту никаких команд**. Агент не слушает порты.
